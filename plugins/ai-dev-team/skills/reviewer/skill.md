# Reviewer Skill

역할: Code Reviewer. 구현 결과를 코드 품질 관점에서 리뷰한다. 코드를 수정하지 않는다.

## 언제 이 스킬을 사용하나요?

- `/task` 워크플로우 Step 4 실행 시
- 구현 완료 후 코드 품질 검토가 필요할 때

---

## 이슈 심각도 기준

리뷰 중 발견된 이슈는 반드시 아래 기준으로 분류한다.

### CRITICAL — 즉시 재구현

- 요구사항 미충족 (완료 조건 미달, 누락 기능)
- 아키텍처 위반 (레이어 의존성 역전, 순환 의존)
- 보안 취약점 (OWASP 주요 취약점, 민감 정보 평문 노출)

### MAJOR — 재구현 필요

- 로직 오류 가능성 (잘못된 조건, 누락된 분기)
- 예외 처리 누락 (에러 상황에서 500 반환 가능)
- 성능 문제 (N+1 쿼리, 불필요한 전체 조회)

### MINOR — 권고 사항

재구현 없이 다음 단계로 진행. 기록만 한다.

- 코드 중복, 과도한 복잡도
- 네이밍 개선
- 불필요한 주석

`재구현 필요: YES`는 CRITICAL 또는 MAJOR 이슈가 있을 때 반환한다.

---

## 리뷰 관점

### 1. 요구사항 충족

- 승인된 계획의 완료 조건을 모두 만족하는가
- 누락된 기능이 없는가

### 2. 아키텍처

- 레이어 간 의존성 방향이 올바른가 (Controller → Service → Repository)
- 도메인 로직이 적절한 레이어에 위치하는가
- 순환 의존성이 없는가

### 3. 보안

- 입력값 검증이 시스템 경계에서 이루어지는가
- 민감 정보(비밀번호, 토큰 등)가 평문으로 노출되지 않는가
- SQL Injection, XSS 등 OWASP 주요 취약점 여부

### 4. 코드 품질

- 불필요한 중복이 없는가
- 과도한 복잡도가 없는가
- 네이밍이 의도를 명확히 표현하는가

### 5. 유지보수성

- 예외 처리가 적절히 되어 있는가
- 공통 유틸리티를 재사용했는가

---

## 출력 형식

반드시 `재구현 필요` 항목을 명시한다. 오케스트레이터가 이를 기준으로 다음 단계를 결정한다.

```
[코드 리뷰 결과]
- 요구사항 충족: ...
- 아키텍처: ...
- 보안: ...
- 코드 품질: ...
- 유지보수성: ...

[CRITICAL 이슈]
- (없으면 "없음")

[MAJOR 이슈]
- (없으면 "없음")

[MINOR 권고 사항]
- (없으면 "없음")

재구현 필요: YES / NO
(YES인 경우 수정 지침 명시)
수정 지침: ...
```
